# AVIOR — PRD 评审意见与聚焦规划

| 项目 | 内容 |
| --- | --- |
| 评审对象 | 《ValiR PRD v0.1》+《R 包验证策略对比分析与实施方案》 |
| 文档版本 | v0.1 |
| 日期 | 2026-07-08 |
| 结论 | 方法论内核扎实，但 PRD 把一份内部落地策略膨胀成了企业级平台；建议砍掉平台形态，聚焦为一个 local-first 的「验证证据编译器」CLI 工具 |

---

## 0. TL;DR

1. **参考文件（策略分析）是可靠的**：双主线模型、intended-for-use 聚焦、GAMP 5 交付物锚定、「自建为主 + 按需采购兜底」的结论都站得住，可以直接作为产品的方法论地基。
2. **PRD 的问题不是细节，是形态**：它把参考文件里「一个团队 2 周跑通的小闭环」放大成了 8 模块、约 40 条 P0、SaaS + 本地双形态、含 21 CFR Part 11 电子签名的平台。这个 MVP 不是 MVP，工作量估计偏差约一个数量级，且最重的部分（签名/托管证据链）恰恰是战略上最不该做的。
3. **聚焦方案**：做一个跑在用户自己环境里的 CLI 工具 —— `avior`。输入 `renv.lock`，输出一份**可签署的验证证据包**（报告 + 决策留痕 + 追溯链 + 环境指纹 + 哈希清单）。证据是文件、留痕靠 git、签署交还客户现有 QMS。先做透第三方包评估这一条线，AI 起草与自研包线作为后续增量。

## 1. 参考文件评审：方法论成立，直接继承

《R 包验证策略对比分析与实施方案》的四个核心判断经得起推敲，聚焦方案全部继承：

| 判断 | 评价 |
| --- | --- |
| 双主线模型（自研包走 SDLC 式验证，第三方包走风险评估式） | ✅ 与 GAMP 5 / R Validation Hub 白皮书一致，是行业共识框架 |
| intended-for-use 聚焦（只深验直接调用包，间接依赖只做版本管理） | ✅ 这是让工作量收敛的关键杠杆，必须作为工具的一等公民 |
| 「自建为主 + 复用开源 + 按需采购兜底」 | ✅ 对预算有限、有开发人力的团队是最优解 |
| 「最小可行起步」（2 周小闭环：SOP 骨架 + 1 个自研包 + 5–10 个第三方包 + 环境冻结） | ✅ 这才是真正的 MVP 定义，应该用它反推产品范围 |

一个需要注意的点：文件中 riskmetric 已转维护、val.meter 演进中的判断，恰恰构成了工具的机会 —— **评估引擎适配层**（今天 riskmetric、明天 val.meter）是社区工具没有、而每个团队都会撞上的真实痛点。

## 2. PRD 评审

### 2.1 成立的部分（保留）

- **问题诊断准确**：「7 个工具手工缝合 + 需求/测试/文档人工黑洞」是真实痛点，直接来自参考文件的一手分析。
- **合规边界画得对**：产品只产证据、不替客户担责、不代签；AI 强制 human-in-the-loop、双署名、全程可审计（FR-E-4）。
- **适配层解耦**（FR-C-6、§9 设计原则）：对冲 riskmetric → val.meter 上游变动，正确且有远见。
- **「小闭环」作为成功标准**（§13）：思路正确 —— 问题在于 PRD 把小闭环当成了平台的**入口**，而它本该定义产品的**全部**。

### 2.2 核心问题（6 条，按严重度排序）

**① MVP 不是 MVP，是平台 1.0。**
8 个模块、约 40 条 P0 需求、12 周交付（§11 Phase 1）。仅模块 G（Part 11 证据链 + 电子签名 + 审计追踪）单独拿出来就是一个独立产品。参考文件的「最小可行起步」是 4 件事、2 周；PRD 把它膨胀了一个数量级，却仍称之为 MVP。按 1–2 人团队的现实产能，这个 Phase 1 需要的不是 12 周而是 12 个月以上。

**② 把 Part 11 电子签名放进 MVP 是最大的战略错误 —— 不只是工作量问题。**
- 目标客户（药企/CRO）**已经有** Part 11 合规的 QMS/eDMS（Veeva、MasterControl、内部 SharePoint 流程等），受控签署与归档发生在那里。重建一套签名系统是在与客户既有采购正面冲突，而不是补缺。
- 更致命的是递归问题：**一旦平台托管证据链与签名，平台自身就成为 GxP 关键系统**。客户 QA 必须对平台做供应商审计，要求平台自身的验证文档、数据完整性与灾备证明。一个初创 MVP 扛不住这个门槛 —— 而 PRD §12 的风险表**完全没有列出「平台自身需要被验证/被审计」这一条**，这是整份 PRD 最大的盲区。
- 正确姿势：产出「**可签署的证据包**」（signature-ready bundle），完整性用哈希清单保证，签署与受控归档交还客户现有体系。合规负担瞬间消失，客户价值几乎不减。

**③ SaaS 形态与目标客户天然冲突。**
评估对象包含客户自研包源码；跑 riskmetric / testthat 意味着在云上执行客户代码；监管证据存放在未经审计的初创云服务上 —— 采购与安全审查过不去。PRD 自己在开放问题 1 里也承认「受监管客户常要求 on-prem」，却仍按双形态设计，交付复杂度 ×2。local-first 的 CLI 从根上绕开整个问题。

**④ AI 定位自相矛盾，且有一处合规硬伤。**
PRD 称 AI 是「差异化核心」，却把它平摊进 8 个模块。更重要的是 **FR-E-1「从源码反推 URS/FS」与参考文件自己强调的 RPVF 原则直接冲突** —— 需求必须先于代码（参考文件 §3.2 / 阶段 3：`vt_use_req` 在代码之前）。从代码反推需求只对**存量包的追溯性（retrospective）验证**成立；对新开发的包，这是审计红旗。PRD 未做此区分，落地时会被客户 QA 打回来。

**⑤ 商业目标脱离行业节奏，北极星指标与自身架构假设打架。**
- G4「MVP 上线 6 个月 ≥ 10 家付费客户」：药企采购周期通常 6–18 个月，合规工具靠口碑冷启动，比一般 SaaS 慢得多。
- 北极星「每月经签署归档的证据包数量」：依赖平台能观测到签署事件；但若签署发生在客户 QMS（见②的正确姿势），平台根本看不到这个数字。指标设计与合规现实互相矛盾。

**⑥ 与免费社区工具正面撞车。**
riskassessment（pharmaR，免费开源）就是组织级评审界面；PRD 模块 C 的评审界面是在重造它。差异化应建在社区**没有**的地方：编排、证据编译、策略即代码、CI 集成 —— 而不是再做一个 Shiny 界面的竞品。

### 2.3 根因

参考文件回答的是「**一个团队**如何落地验证体系」；PRD 未经需求验证就跳到了「为**所有团队**造平台」。诚实的顺序应该是：先把自己团队的真实项目当第一个用户，用工具把参考文件的阶段 0–2 自动化跑通，再谈产品化。

## 3. 聚焦方案：AVIOR，验证证据编译器

### 3.1 一句话定位

> **`renv.lock` 进，可签署的验证证据包出。**
> 一个跑在你自己环境/CI 里的 CLI 工具，把「依赖识别 → 风险评分 → 决策留痕 → 定向测试 → 环境指纹 → 审计就绪报告」编译成一条命令的产出。像 pytest / terraform 一样活在仓库里，而不是像平台一样把你搬进去。

（命名：AVIOR 即船底座 ε 星，位于神话中阿尔戈号的**龙骨** —— 「验证的龙骨」，寓意恰好；CLI 名 `avior` 也顺手。ValiR 本就是暂定代号。）

### 3.2 三条设计公理（所有裁剪决策的依据）

| 公理 | 含义 | 直接后果 |
| --- | --- | --- |
| **做管线，不做平台** | CLI 工具，跑在用户环境与 CI 里；无服务端、无数据库、无多租户 | 砍掉 SaaS/本地双形态、RBAC/SSO、托管基础设施 |
| **证据即文件** | 一切产物（策略、决策、测试、报告）是版本库中的文件；git 历史即变更留痕，PR 即评审流 | 砍掉自建审计追踪系统；审计追踪 = git log + 哈希清单 |
| **签署留在客户 QMS** | 产出 signature-ready 证据包（含 SHA-256 manifest 保证防篡改），签署/受控归档走客户现有体系 | 砍掉电子签名与 Part 11 认证负担；平台自身不再是 GxP 系统 |

### 3.3 MVP 范围：先做透一条线

**MVP = 第三方包风险评估管线**（对应参考文件阶段 1–2、PRD 模块 B+C 的内核）。理由：

1. **受众最大**：所有受监管 R 团队都有第三方包问题；自研包不是每家都有。
2. **自动化率最高**：除人工决策外全部可机器完成，工具价值密度最高。
3. **空白真实存在**：riskassessment 解决的是「界面评审」，没解决「CI 原生 + 证据编译 + 策略即代码」。
4. **自研包线已有 valtools**：直接用即可，MVP 阶段包一层壳增值有限（V2 再编排它）。

### 3.4 CLI 设计

```text
avior init      # 生成 avior.yml —— SOP 即代码：适用范围、三分类规则、
                # riskmetric 权重与阈值（含"理由"必填字段，供审计辩护）
avior scan      # 解析 renv.lock / DESCRIPTION：
                # base/recommended vs contributed vs custom 三分类
                # + intended-for-use 直接调用识别（直接 vs 间接依赖）
avior assess    # 经适配层跑 riskmetric（未来 val.meter），按策略聚合风险分
avior review    # 为需人工决策的包生成决策记录桩（YAML）；
                # 校验完整性：每个在册包必须有已署名决策
avior test      # 运行中高风险包的定向 testthat 测试，结果入证据
avior bundle    # 编译证据包：报告（Word/PDF/HTML）+ 追溯链
                # （包 → 评分 → 决策 → 测试 → 环境）+ 环境指纹
                # + manifest（SHA-256）→ 不可变 zip
avior check     # CI 门禁：证据与当前 lockfile 的漂移检测；
                # 有包无决策 / 评分过期 / 环境变更 → 红
```

`avior check` 是隐藏的杀手锏：放进 CI 后，**持续验证（PRD 模块 F 的 80%）自动获得** —— 依赖一变，门禁就红，无需自建监控引擎与触发系统。

### 3.5 项目内文件布局与证据包结构

> 注：本节为讨论期示意，目录与文件命名（如 `bundle-2026-07-08/`、`report.pdf`）以《AVIOR PRD》§6.4 冻结契约为准：`bundle-<UTC 时间戳>`（如 `bundle-20260708T120000Z`）、`report.html/docx`。

```text
your-r-project/
├── renv.lock
└── validation/
    ├── avior.yml            # 策略即代码（受控文档，进 git，改动走 PR）
    ├── decisions/           # 每包一条决策记录
    │   └── dplyr.yml
    ├── tests/               # 中高风险包的定向 testthat 测试
    │   └── test-survival-coxph.R
    └── evidence/            # 产出的不可变证据包
        └── bundle-2026-07-08/
            ├── report.pdf / report.docx
            ├── traceability.csv
            ├── environment.json    # R 版本、平台、快照 ID、renv.lock 哈希
            ├── decisions/          # 决策记录快照
            └── MANIFEST.sha256     # 全部文件哈希 → 防篡改
```

策略与决策记录示意：

```yaml
# validation/avior.yml —— SOP 即代码
policy:
  risk_thresholds: { low: 0.25, high: 0.55 }
  rationale: "阈值参考 riskassessment 默认并经统计编程组 2026-07 评审会确认"
  weights:
    has_vignettes: 0.5
    covr_coverage: 2.0
scope:
  intended_for_use: auto        # 从 DESCRIPTION/library() 调用自动识别，可人工覆盖
  exclude: [datasets]
depth_by_risk:                  # 风险分级 → 测试深度对应表
  low: metadata_only
  medium: smoke_tests
  high: targeted_tests
```

```yaml
# validation/decisions/dplyr.yml —— 决策留痕
package: dplyr
version: 1.1.4
score_snapshot: 0.18            # 引用哪次评分
decision: include               # include / include_with_tests / exclude
rationale: "低风险；核心数据操作，团队 5 年使用史"
reviewed_by: "jin@example.com"
date: 2026-07-08
ai_assisted: false              # AI 参与起草时必须为 true 并附人工确认者
```

git 提交历史 + PR 评审 + manifest 哈希，合起来就是 who/what/when/why 的审计追踪 —— 不需要自建一套。

### 3.6 架构（薄得刻意）

```text
┌──────────────────────────────────────────────┐
│ CLI（avior）：init/scan/assess/review/test/  │
│               bundle/check                   │
├──────────────────────────────────────────────┤
│ 核心 R 包：策略引擎 · 三分类/依赖分析 ·      │
│            决策校验 · 追溯链组装             │
├──────────────┬───────────────┬───────────────┤
│ 评估适配层    │ 测试运行器     │ 报告编译器     │
│ riskmetric → │ testthat/covr │ Quarto →      │
│ val.meter    │               │ PDF/Word/HTML │
├──────────────┴───────────────┴───────────────┤
│ 状态 = 仓库里的文件（无服务端、无数据库）      │
└──────────────────────────────────────────────┘
```

技术选型：核心用 R 包实现（用户是 R 团队，riskmetric 是 R，零额外运行时），CLI 入口用 `Rscript` 包装；报告用 Quarto；配置/决策全部 YAML。

### 3.7 明确不做（MVP 阶段的纪律）

| 不做 | 替代 |
| --- | --- |
| 电子签名 / Part 11 认证 | signature-ready 证据包 + 哈希清单，签署走客户 QMS |
| SaaS / 托管服务 / Web 评审界面 | local-first CLI；证据包内含静态 HTML 看板（是文件，不是服务） |
| 自建审计追踪与用户体系 | git 历史 + PR 评审 + manifest |
| 自研包 SDLC 验证 | V2 编排 valtools；当下直接用 valtools 本身 |
| 监控/触发引擎（diffify、oysteR） | `avior check` CI 门禁覆盖大部分需求；完整版 V2 |
| SOP 文档管理、多项目治理、双语模板 | V2+；`avior.yml` 的 rationale 字段先扛住审计辩护 |
| AI 起草（首版） | V1.1 第一优先级，见 §4 —— 先建轨道，再上副驾 |

## 4. 路线图（按 1–2 人产能的现实版）

| 阶段 | 时间 | 内容 | 出口标准 |
| --- | --- | --- | --- |
| **W1–2** | 定格式 | 证据包结构 + `avior.yml` schema 定稿；拿自己团队的真实项目当 design partner | 一份手工组装的样例证据包，QA 同事看过说"这个能签" |
| **W3–6** | 核心管线 | `scan` / `assess`（适配层）/ 策略引擎 / `review` 决策校验 | 真实项目 50 包出评分与决策桩 |
| **W7–9** | 证据编译 | `bundle`（Quarto 报告 + 追溯链 + manifest）+ `check` CI 门禁 | CI 中跑通红/绿门禁 |
| **W10–12** | 试点打磨 | 真实项目端到端；打磨报告模板；写方法论文档（引用白皮书四准则 + GAMP 5，给审计员看的） | 一份真实证据包通过内部 QA 审查 |
| **W12 决策点** | — | 依试点反馈选 V1.1 方向：AI 起草定向测试 vs 自研包线（valtools 编排） | — |

**V1.1（推荐首选）**：`avior draft tests <pkg>` —— AI 针对「声明的用途」起草定向 testthat 测试，产物强制标注 AI 生成、决策记录双署名（对应 PRD FR-E-2/E-4，收窄到最可行的一个场景）。AI 放在管线之后而非之前的原因：**AI 产物需要结构化的落点**（决策记录、测试脚手架、追溯链），先有轨道，副驾才有座位。

**V2**：自研包线（编排 valtools，AI 起草需求仅限存量包追溯性验证并显式标注）、diffify/oysteR 监控触发、NMPA 中英双语模板包。

**商业化（轻量提示，非当前重点）**：open-core —— CLI 开源换取保守行业的信任与采纳；收费层 = 模板包（双语/行业化）、AI 起草、企业支持。开源恰好呼应 PRD 自己的「开源友好、反锁定」护城河主张。

| PRD 模块 | 处置 | 去向 |
| --- | --- | --- |
| A 项目与 SOP 引擎 | **收缩** | `avior.yml`（策略即代码）+ rationale 字段；SOP 文档管理砍掉 |

## 6. 成功标准（替换 PRD §13）

1. 一个真实项目（约 50 个第三方包）从 `renv.lock` 到可签署证据包 **≤ 1 个工作日**（对照：参考文件阶段 0–2 的人工路径按周计）。
2. 证据包**通过一次真实的内部 QA 审查**，审查意见回灌模板。
3. **可重现性**：同一输入重跑，产出 manifest 一致的证据包。
4. `avior check` 在 CI 中稳定运行 ≥ 4 周，漂移检出无漏报。

北极星指标相应替换为：**「通过完整性校验的证据包生成数」**（工具可观测），而非「经签署归档数」（工具观测不到）。

## 7. 风险与开放问题

| 风险 | 缓解 |
| --- | --- |
| riskmetric 转维护 / val.meter 迁移 | 适配层（继承 PRD 正确设计）；锁定 riskmetric 版本入证据包 |
| 「git 即审计追踪」能否被客户 QA 接受 | W1–2 用样例证据包先行验证；不行则在 bundle 内附「变更历史导出」章节（仍是文件，不是系统） |
| intended-for-use 自动识别准确率 | 允许人工覆盖清单；识别结果本身进决策记录 |
| 50+ 包批量评分耗时 | 评分缓存（按包名+版本+引擎版本），只重算变更项 |
| 单人/小团队产能 | 公理三条已把范围钉死；任何新需求先过「是管线还是平台」的判断 |

开放问题（保留 PRD 中仍然有效的两个）：
1. AI 起草用第三方大模型时，客户源码隐私边界如何设定？（V1.1 前必须回答；local-first 架构下可选本地模型作为兜底）
2. 正式名称：建议直接用 **AVIOR / `avior`**，放弃 ValiR。
