# **1  执行摘要**

结论先行：在你的约束下（申报级 GxP、自研与第三方包并存、有开发人力但预算有限），推荐采取**「以开源框架自建为主、商业方案作为高风险统计包兜底」的混合策略**，而非纯自研或纯采购。

核心判断：这四家并不是同一层面的竞品，而是三种不同「物种」——**标准/框架派**（R Validation Hub \+ PHUSE valtools，免费、需自建）、**商业「验证包即服务」派**（Atorus 的 OpenVal，订阅制买现成）、**咨询\+自研工具派**（Jumping Rivers 的 Litmusverse，服务为主）。

需要先澄清一个常见误解：**你列出的「openval」就是 Atorus Research 的商业产品 OpenVal 本身**，二者并非并列的两个组织。

落地建议的一句话版本：用 valtools 建立自研包的验证框架（RPVF），用 riskmetric / riskassessment 做第三方包的风险分级评估，用 Posit Package Manager \+ renv 冻结环境，用 diffify / oysteR 做持续监控；对少数高风险统计计算包，若人工深度测试成本过高，再考虑向 OpenVal 采购单包验证或请 Jumping Rivers 做专项咨询。

### **为什么不是纯自建，也不是纯采购**

- **纯自建成本被低估：**从零写风险评分引擎、维护指标口径、跟踪 CRAN 版本变化，长期维护负担重；社区已有成熟工具（riskmetric、valtools）覆盖 80% 的机械性工作，重复造轮子不划算。

- **纯采购锁定与合规主体问题：**OpenVal 能省人力，但订阅费持续支出、验证边界受供应商定义、自研包无法外包，且监管责任主体始终是申办方自己，供应商文档只是证据之一。

- **混合最贴合你的画像：**有人力→能吃下开源框架的搭建与运维；预算有限→把钱花在刀刃上（仅高风险包按需采购）；自研\+第三方并存→正好对应 valtools（自研）与 riskmetric（第三方）两条互补主线。

# **2  背景与合规框架：先对齐「验证到底在验什么」**

在对比工具之前，必须先厘清 R 包验证的监管逻辑，否则容易把「工具选型」误当成「合规策略」。工具只是产出证据的手段，合规主体和判断标准始终在申办方一侧。

## **2\.1  为什么 R 包在申报环境必须验证**

监管机构（FDA、EMA、NMPA）并不禁止开源软件用于申报，但要求任何用于产生申报结论的软件都有证据表明其「按预期工作、结果准确可复现」。R 语言与包本身合法可用，关键在于申办方能否为所用的包建立与风险相称的验证证据链。数据完整性（ALCOA\+）、可追溯性、可复现性是三条底线。

## **2\.2  三个必须先建立的概念**

### **① 两条验证主线：软件开发式 vs 风险评估式**

- **软件开发式验证（software\-based）：**把包当软件产品，走完整 SDLC——需求（URS/FS）→设计→实现→测试→追溯矩阵→报告。主要用于**自研包**，因为你拥有源码与需求定义权。

- **风险评估式验证（risk\-based）：**对无法控制其开发过程的**第三方开源包**，通过评估「用途、维护活跃度、社区使用度、测试覆盖」等间接证据给出风险分级，再据风险高低决定测试深度。

### **② 「Intended for use」聚焦原则**

R Validation Hub 白皮书的关键主张：风险评估应聚焦于用户直接调用的包（intended\-for\-use），而非其所有间接依赖（Imports）。对直接调用包的功能验证，本身就间接测试了它所依赖的底层包。依赖只需从可复现性角度做版本管理，不必逐一做准确性验证。这条原则直接决定了你的验证工作量能否收敛。

### **③ 包的三分类与差异化要求**

|**包类型**|**特征**|**验证要求**|
|---|---|---|
|Base / Recommended（核心包）|R Foundation 开发并随 R 发行，SDLC 严格|视为低风险，一般无需额外补充验证|
|Contributed（第三方开源）|CRAN/GitHub 等，任何人可发布，仅过基本技术检查，准确性无保证|按风险分级：低风险轻量、中高风险需针对用途补充单元测试（testthat）|
|Custom（自研）|为特定用途定制、使用者面窄|最需严格软件开发式验证；受众小反而更应重视|

## **2\.3  可对齐的指导文件锚点**

- **R Foundation《R: Regulatory Compliance and Validation Issues》**（2018）——R 用于受监管临床环境的基础性指导。

- **R Validation Hub 白皮书**《A Risk\-based Approach for Assessing R Package Accuracy within a Validated Infrastructure》（2020）——风险评估四准则的来源。

- **ISPE GAMP 5（第二版，2022）**——计算机化系统生命周期最小交付物：规范（Specifications）、风险评估（RA）、测试（Tests）、追溯矩阵（Traceability Matrix）；软件按类别分级验证的思想与风险评估法一致。

- **FDA CSA（Computer Software Assurance）思路**——把有限验证资源投向高风险功能，与「风险分级」完全同向。

- **TransCelerate MSA、Posit/RStudio 合规文档**——环境（Workbench / Package Manager / Connect）层面的合规证据。

- **NMPA 视角**——《药物临床试验数据管理与统计分析计划》《计算机化系统》相关要求与数据完整性原则；若涉及国内申报，同一套证据链（RA \+ 测试 \+ 追溯 \+ 变更控制）通常可复用，但需以中文 SOP 固化并保留签署记录。

# **3  四家 / 工具逐一剖析**

## **3\.1  R Validation Hub（pharmaR）——开源标准与风险评估引擎**

**定位：**由 PSI 的 AIMS SIG 于 2018 年发起、现挂靠 R Consortium 的跨行业非营利协作组织，使命是让 R 能用于生物制药监管申报。它提供的是**方法论 \+ 开源工具**，不卖服务。

**产出物：**① 白皮书（风险评估四准则：用途/类型、维护、社区使用、测试）；② **\{riskmetric\}** R 包——抓取包元数据并对开发实践、文档、社区参与、可维护性等约 20\+ 指标打分，聚合为 0–1 风险分（现转入仅维护状态）；③ **\{riskassessment\}** Shiny 应用——在 riskmetric 之上提供组织级评审界面、指标加权、人工评级、阈值决策与可下载报告；④ 新旗舰 **\{val\.meter\}**——官方正把开发重心迁移到它，作为下一代评估工具。

**重要边界：**riskmetric 明确**不评估「准确性」与「影响」**——它衡量的是「代码质量/风险的间接信号」，而某个函数算得对不对、算错了后果多大，必须由组织自行通过针对用途的测试和人工判断来确定。这是所有纯自动化打分工具的共同天花板。

|**优点**|**缺点/注意**|
|---|---|
|- 免费、开源、社区共识背书，最贴合监管期望的「风险分级」范式<br>- 指标可编程、可批量、可复现，天然产出结构化证据<br>- riskassessment 提供组织级评审留痕与阈值规则，接近可直接用于 GxP 决策<br>- 有白皮书作为方法论出处，审计时可引用|- 只解决第三方包「风险打分」，不覆盖自研包的完整 SDLC 验证<br>- 不判断准确性，仍需大量人工测试补齐高风险包<br>- 指标权重/阈值需组织自行定义与固化，前期投入不小<br>- riskmetric 转维护、val\.meter 尚在演进，需跟踪版本与迁移路径|

## **3\.2  PHUSE valtools——自研包的验证框架（RPVF）脚手架**

**定位：**由 PHUSE「R Package Validation Framework」工作组开发的 R 包，是对 usethis/devtools 标准开发流程的**「螺栓式」补充**，把验证基础设施直接嵌进包的 vignettes/validation 目录。所有函数以 vt\_ 前缀开头。

**工作流：**vt\_create\_package\(\) 建带验证骨架的新包 → vt\_use\_validation\(\) 给现有包加验证基建 → vt\_use\_req\(\) 写需求 → vt\_use\_test\_case\(\) / vt\_use\_test\_code\(\) 写测试用例与测试代码 → vt\_use\_change\_log\(\) 变更日志 → vt\_use\_report\(\) 生成验证报告 Rmd。报告编译时自动生成**需求—测试—追溯矩阵**，正好对应 GAMP 5 的最小交付物。

**四种验证时点（灵活适配不同 SOP）：**Validation on Source（源码）、on Build（编译成不可变 tar）、on Install（安装到目标机时重跑）、After Install（环境更新后随时再验证）。这让「验证证据跟随包分发」成为可能。

|**优点**|**缺点/注意**|
|---|---|
|- 专为自研包设计，直接产出需求/测试/追溯/报告全套 GxP 文档<br>- 与 R 原生开发工具链无缝衔接，学习曲线平缓<br>- 报告可复现、可版本化，验证随包走<br>- 有 FredHutch 公开教程与社区案例可借鉴|- 主要面向「你拥有源码」的包，不解决第三方包风险评估<br>- 报告编译可能耗时（含全部测试代码运行，复杂包可达数十小时）<br>- 存在社区 fork 修 bug 的情况，需锁定可信版本<br>- 需求/测试要人来写，工具只是脚手架而非替你验证|

## **3\.3  Atorus Research / OpenVal——商业「验证包即服务」**

**定位：**OpenVal 是 Atorus 的**订阅制商业产品**：一个不断增长的已验证 R 包仓库（约 300\+ 且持续扩充）\+ 验证文档 \+ R 环境交付系统，采用**自动化 \+ 人工混合测试**。可按需提交缺失包的验证请求，验证成果在客户间共享复用。

**方法论：**与 R Validation Hub 同源（也引用四准则），但补充了强人工判断——例如指出「80% 代码覆盖率≠关键函数被测到」，强调人工审查关键统计函数。可搭配 Posit Package Manager 把已验证包发布为内部「Validated」仓库。

|**优点**|**缺点/注意**|
|---|---|
|- 买现成，最省人力与前期投入，上手快<br>- 混合验证含人工判断，覆盖自动打分的盲区（准确性）<br>- 包与文档在客户间摊薄成本，社区效应带来更多已验证包<br>- 适合小 biotech 快速起步或大厂快速铺开|- 持续订阅费；预算有限者长期成本需算清<br>- 验证边界由供应商定义，未必完全匹配你的用途与 SOP<br>- 无法覆盖你的自研包；供应商文档是证据之一而非合规责任转移<br>- 存在供应商锁定与可迁移性风险|

## **3\.4  Jumping Rivers（Litmusverse）——咨询 \+ 自研工具箱**

**定位：**一家数据科学咨询公司，长期做 R 包验证，自研了 **Litmusverse** 工具箱（含 Litmus\.dashboard——交互式浏览包级风险分并跟踪随时间变化）。底层同样整合社区工具：riskmetric（风险打分）、oysteR（CVE 安全漏洞扫描）、diffify（跨版本差异比对）。也提供 Shiny 应用验证、健康检查等服务。

|**优点**|**缺点/注意**|
|---|---|
|- 服务\+工具结合，能补齐团队方法论与人力缺口<br>- 覆盖安全（oysteR）与版本变更（diffify）等常被忽略的维度<br>- 把 Shiny 应用当包来验证的思路，对交互式工具很实用<br>- 适合需要外部专家背书或阶段性加速的团队|- 以咨询/服务为主，成本模式偏项目制，非纯开源自助<br>- Litmusverse 部分为其自研，长期依赖存在耦合<br>- 对「自建为主」的团队，更适合作为专项外援而非主干<br>- 能力与交付质量依赖具体合作范围|

### **3\.5  配套（非同类但落地必备）**

- **Posit Package Manager（PPM）**：冻结 CRAN/Bioconductor 快照、建内部「Validated」仓库、基于 osv\.dev 做漏洞扫描——环境层可复现性的基石。

- **renv**：项目级依赖锁定，保证 R 版本与包版本可还原。

- **thevalidatoR**：另一款自动生成验证报告的工具，可与 valtools 互为参考。

# **4  横向对比矩阵**

下表按关键决策维度并排四家。注意 R Validation Hub 与 valtools 高度互补（前者管第三方评估、后者管自研验证），实际自建方案通常两者一起用。

|**维度**|**R Validation Hub**|**valtools \(PHUSE\)**|**Atorus / OpenVal**|**Jumping Rivers**|
|---|---|---|---|---|
|**物种**|开源标准\+评估引擎|开源验证框架脚手架|商业验证包订阅|咨询\+自研工具|
|**主要覆盖包类型**|第三方开源包|自研/自控源码包|第三方（含部分）|第三方 \+ Shiny 应用|
|**开源/商业**|免费开源|免费开源|订阅收费|服务收费|
|**核心产出**|风险分\+评审报告|需求/测试/追溯/报告|已验证包\+文档\+环境|风险看板\+咨询交付|
|**自动化程度**|高（可编程批量）|中（脚手架\+需人写）|高（自动\+人工）|中高（工具\+人工）|
|**人工判断**|需组织自行补|需组织自行写|供应商已含|供应商/顾问含|
|**准确性验证**|不含，需自补|含（你写测试）|含（人工审查）|含（服务范围内）|
|**前期投入**|中（定阈值/权重）|中（建框架/写测试）|低（买现成）|低\-中（外包）|
|**长期成本**|低（自维护）|低（自维护）|持续订阅|项目制/持续|
|**锁定风险**|低|低|中\-高|中|
|**合规责任主体**|申办方|申办方|仍是申办方|仍是申办方|
|**最适合**|第三方包评估主干|自研包验证主干|快速起步/兜底|专项外援/加速|

## **4\.1  相同点**

- 四者都认同「风险分级」而非「一刀切全验」，都追溯到 R Validation Hub 白皮书的四准则。

- 都强调可复现环境（PPM / renv / 冻结快照）是验证的前提。

- 都承认纯自动化打分不足以证明准确性，人工判断不可或缺。

- 合规责任主体始终是申办方，工具/供应商只提供证据。

## **4\.2  不同点（一句话区分）**

- **R Validation Hub** 回答「这个第三方包有多大风险」；

- **valtools** 回答「我自己的包如何走完验证并留档」；

- **OpenVal** 回答「能不能直接买到已验证的包，别自己折腾」；

- **Jumping Rivers** 回答「能不能请人帮我把这件事做好、并给我工具」。

# **5  「自建 vs 复用」决策分析**

这不是二选一，而是「哪些自建、哪些复用」的组合题。下表把决策拆到可操作粒度。

|**工作模块**|**建议做法**|**理由**|
|---|---|---|
|**方法论/风险准则**|复用（R Validation Hub 白皮书）|现成、社区共识、审计可引用，无需自创|
|**第三方包风险打分**|复用 riskmetric \+ riskassessment|机械性工作已被工具解决，自研不划算|
|**组织阈值/权重/评审 SOP**|自建|必须体现你的用途与风险偏好，无法外包|
|**自研包验证**|复用 valtools 框架 \+ 自建测试内容|框架复用、需求与测试用例只能你写|
|**环境与可复现**|复用 PPM \+ renv|行业标准做法，自建成本高且无收益|
|**持续/变更验证**|复用 diffify \+ oysteR \+ 自建触发规则|工具成熟，规则需嵌入你的变更控制|
|**高风险统计包准确性测试**|自建为主，必要时向 OpenVal/JR 采购|预算有限先自做；成本过高的少数包再买|

## **5\.1  针对你的推荐方案**

**推荐：**开源自建为主干（valtools 管自研、riskmetric/riskassessment 管第三方、PPM/renv 管环境），商业方案作为**按需兜底**。这样把有限预算集中在「少数高风险统计包的深度验证」上，其余用免费工具\+自有人力覆盖，既满足申报级要求，又控制长期成本、避免锁定。

**何时应触发采购/外援：**当某个高风险统计计算包（如复杂建模/生存分析/PK 包）自行做准确性金标准比对的成本，明显高于购买单包验证或专项咨询时；或临近关键申报里程碑、时间不允许自建时。

# **6  实施方案（分阶段路线图）**

## **阶段 0（第 1–3 周）：治理与 SOP 奠基**

1. 制定《R 包验证策略 SOP》：明确适用范围、包三分类、风险分级规则、riskmetric 指标权重与风险阈值、纳入/排除 GxP 环境的判定规则。

2. 定义风险分级与测试深度对应表（低/中/高 → 轻量/中等/深度测试）。

3. 建立角色与 RACI：验证负责人、开发者、独立评审、QA/合规审批、IT/环境管理。

4. 确定交付物模板（对齐 GAMP 5）：URS/FS、风险评估、测试用例/结果、追溯矩阵、验证报告、变更日志。

**交付物：**SOP v1、风险分级规则表、RACI、文档模板库。

## **阶段 1（第 3–6 周）：环境与可复现基建**

1. 部署/配置 Posit Package Manager，建立冻结的 CRAN/Bioconductor 快照与内部「Validated」仓库。

2. 在项目中引入 renv，锁定 R 版本与包版本；建立环境快照的版本化与留存机制。

3. 开启 PPM 的漏洞扫描（osv\.dev），与 oysteR 形成安全基线。

**交付物：**受控仓库、环境基线文档、可复现性验证记录。

## **阶段 2（第 5–9 周）：第三方包风险评估流水线**

1. 圈定「intended\-for\-use」直接调用包清单（不逐一验证间接依赖）。

2. 用 riskmetric 批量打分：pkg\_ref\(\) → pkg\_assess\(\) → pkg\_score\(\)，按 SOP 权重聚合。

3. 在 riskassessment 应用中做人工评级、留痕评论、按阈值给出「纳入/需补测/排除」决策。

4. 对中高风险包，针对用途补写 testthat 测试；导出并归档风险评估报告。

**交付物：**直接调用包清单、风险评分与评级报告、补充测试、纳入决策记录。

## **阶段 3（第 8–14 周）：自研包 RPVF 验证**

1. 对每个自研包用 valtools 加验证基建：vt\_use\_validation\(\)。

2. 先写需求（vt\_use\_req，代码之前），再写测试用例与测试代码（vt\_use\_test\_case / vt\_use\_test\_code）。

3. 用 vt\_use\_report 生成验证报告，编译自动产出需求—测试—追溯矩阵。

4. 按 SOP 选定验证时点（建议 on Build 或 on Install，保证验证证据随包不可变分发）。

5. 接入 CI/CD（如 GitHub Actions）：提交即跑 R CMD check \+ testthat \+ 覆盖率，报告归档。

**交付物：**每个自研包的验证报告包（需求/测试/追溯/变更日志）\+ CI 流水线。

## **阶段 4（持续）：持续验证与变更控制**

1. 版本变更监控：用 diffify 比对包新旧版本差异，评估是否触发再验证。

2. 安全监控：定期 oysteR / PPM 漏洞扫描，命中即走变更评估。

3. 定期再评估：按周期或「升级/环境变更/依赖变更」事件重跑 riskmetric 与自研包验证报告。

4. 所有变更纳入变更控制流程并留签署记录（国内申报需中文归档）。

**交付物：**持续验证台账、变更控制记录、周期性再评估报告。

# **7  实施风险与缓解**

|**风险**|**影响**|**缓解**|
|---|---|---|
|**riskmetric 转维护、val\.meter 演进中**|工具迁移、指标口径变动|锁定当前版本；跟踪 val\.meter；SOP 与具体工具解耦，只绑定「四准则」|
|**阈值/权重设定主观**|审计质疑一致性|阈值来源写入 SOP，保留论证依据；跨团队评审达成共识|
|**自研包验证报告编译耗时**|里程碑延误|测试分层、并行化、CI 缓存；关键包优先|
|**把工具选型当合规策略**|证据链断裂|始终以「用途→风险→测试深度→留档」为主线，工具仅产证据|
|**国内/国际申报要求差异**|重复劳动|同一证据链设计为可复用；中文 SOP 与签署记录并行维护|
|**过度验证或验证不足**|成本失控 / 合规缺口|严格执行 intended\-for\-use 聚焦与风险分级|

# **8  结论与下一步**

**结论：**借鉴他们的思路「自建为主 \+ 复用其工具 \+ 按需采购兜底」，是你当前约束下性价比最高、合规最稳、锁定最小的路径。不要纯自研（重复造轮子），也不要纯采购（成本与锁定，且覆盖不了自研包）。

**最小可行起步（建议未来 2 周先做）：**

1. 起草《R 包验证策略 SOP》骨架 \+ 风险分级规则表。

2. 选 1 个真实自研包跑通 valtools 全流程，产出一份样例验证报告。

3. 选 5–10 个高频第三方包跑通 riskmetric \+ riskassessment，校准你的阈值。

4. 用一个真实包验证 PPM \+ renv 的环境冻结与还原。

跑通这套「小闭环」后，再横向铺开到全部包，并把高风险少数包的采购/外援决策留到那时按实际成本判断。
