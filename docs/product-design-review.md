## 0. TL;DR

1. **形态与定位判定成立**：从 ValiR 平台版收敛到 local-first 证据编译器 CLI 的裁剪逻辑（design-review §2.2 六条问题）在两份调研文档中都能找到一手依据，三条公理至今没有被任何后续需求击穿，范围纪律执行得好。
2. **最大的问题不在方向，在数据契约的完备性**：字节级可复现（NFR-1、各命令 AC）目前只有排序规则（FR-X-7）撑着，缺规范化序列化规范；样例证据包里已经出现了 PRD 未定义的枚举值（`version_managed`）；环境指纹缺了调研文档自己反复强调的字段（BLAS/LAPACK、locale）。这些都是 M0「格式定稿」的正事，应在冻结前落档。
3. **路线可行但有三处需要提前拆弹**：默认权重里的 `covr_coverage`（昂贵指标）与 assess 30 分钟 AC 有正面冲突，W3 就该实测而不是 W6；报告模板（Quarto → docx）历来是打磨黑洞，建议 HTML 先行；三平台字节一致是横切需求，golden 测试基建必须 M1 第一周就位，无法后补。

## 1. 方向判定：做对了什么（简述，不展开）

| 判断 | 依据 |
| --- | --- |
| 「证据编译器而非平台」的收敛正确 | 调研文档 §7.1 的行业结论是「复用而非自建、无同行从零造平台」；design-review §2.2-② 对「平台自身成为 GxP 关键系统」递归问题的识别，是整个收敛中最有价值的一刀 |
| 三条公理作为需求准入检验有效 | PRD v1.1→v1.5 的五次修订全部是澄清与收紧，没有一次范围扩张；Q1–Q4 决策均与公理自洽 |
| manifest 信任边界的诚实表述（§5.8） | 主动写明「manifest 不是信任根、锚点在外部」，比宣称防篡改更能通过 ALCOA+ 评审——这是审计辩护意识的体现 |
| M0 格式先行 + 手工样例 | 「QA 说这个格式能签」是全项目最大的单点假设，用两周手工样例去验证它，是正确的最小代价探针 |
| 引擎适配层 + 版本钉入证据 | 直接对冲 riskmetric 维护模式/val.meter 演进这一上游最大不确定性 |
| §1.4 验证对象边界（项目级证据、不认证单包、不验分析代码） | 与白皮书「用途决定风险档」的方法论一致，堵住了「此包已验证」这种审计上站不住的表述 |

以下不再重复优点，集中说问题。

## 2. 问题清单

按严重度分三级：**A = M0 冻结前必须处理**（契约级，事后改是破坏性变更）；**B = M1 开工前应决策**（设计级）；**C = 路线级，持续关注**。

### A 级：契约缺口，趁 schema 未冻结修掉

**A1｜「字节级一致」缺规范化序列化规范——FR-X-7 只解决了一半。**
排序定了，但同样影响字节一致的还有：浮点数格式（riskmetric 返回 double，`0.61` 还是 `0.6100000000000001` 还是 `6.1e-01`？）、YAML 风格（block vs flow、缩进、引号策略、键序）、编码与换行（三平台 CI 下 CRLF 是现实风险）、时间戳格式。这些不定，NFR-1 与各命令的「重复执行输出字节一致」AC 在三平台上必然间歇性失败，而且是最难排查的一类失败。
**建议**：新增 **FR-X-8「规范化序列化」**：全部生成物 UTF-8 无 BOM、LF 换行；数值统一定点格式（如评分四位小数，`sprintf("%.4f")` 级别的明确规则）；YAML 生成风格固定（缩进、引号、flow/block 逐产物指明）；时间戳一律 UTC ISO-8601 秒级。实现上先写「规范化写出器」工具函数，一切产物经它落盘——这是 M1 第一批代码。

**A2｜样例与 PRD 契约不一致：`version_managed` 不存在于任何 schema 定义。**
`examples/.../traceability.csv` 中 transitive 行（minqa）的 decision 列填了 `version_managed`，但 FR-REVIEW-2 的决策枚举只有 `include / include_with_tests / exclude`，§6.5 也只说「无决策文件的行两列均留空」，未定义 decision 列在 transitive 行的取值。样例即将拿去给 QA 做 M0 格式评审，评审通过的格式却和 PRD 对不上，会把不一致固化进冻结契约。
**建议**：二选一并同步双方——(a) transitive 行 decision 列留空，处理方式放 notes；(b) 在 §6.5 正式定义行级状态值 `version_managed`（仅限 transitive 行，与决策枚举分开命名空间）。倾向 (b)：审计员读 CSV 时，一个明确的「仅版本管理」状态比空白更可辩护，这也正是 intended-for-use 原则在追溯矩阵上的显式表达。

**A3｜环境指纹缺了调研文档自己强调的关键字段。**
调研文档 §4.4 把「数值可复现性依赖 BLAS/LAPACK 实现与线程」列为合规报告的关键技术点，§3.3 强调 locale 排序与 `sessionInfo()` 留档，§4.3 强调容器按 SHA256 摘要锁定。而 FR-BUNDLE-5 / 样例 `environment.json` 只记录 R 版本、平台、仓库 URL、lockfile 哈希、引擎版本——恰好漏掉了自家调研认定最重要的那几项。这是「证明环境可复现」的产物，字段却不足以重建环境。
**建议**：扩展 FR-BUNDLE-5：增加 `locale`（至少 LC_COLLATE）、BLAS/LAPACK 信息（R ≥ 4.1 可经 `sessionInfo()`/`La_library()` 获取，取不到记 `unknown` 而非省略）、容器镜像摘要（检测到容器环境时记录，如从环境变量/cgroup，检测不到记 null）、完整 `sessionInfo()` 文本作为 bundle 附件。成本极低，审计价值高——ADRG 实践里这些正是审评员要的东西。

**A4｜测试证据与运行环境脱钩。**
FR-TEST-2 规定 `test-results.yml` 记录 testthat 版本与被测包版本，但不记录**测试运行时的环境**（lockfile 哈希 / 环境指纹）。若有人在未 `renv::restore()` 的库里跑了测试，结果照样入档、`check` 照样绿——证据链在「测试」这一环对环境不设防，而这恰是四准则之外 GAMP 5 最看重的测试证据有效性。
**建议**：`test-results.yml` 增加 `lockfile_sha256`（运行时点）与 R 版本/平台字段；`check` 增加一条规则：测试结果的 lockfile 哈希 ≠ 当前 inventory 基准 → 测试过期（复用 FR-TEST-2 已有的「过期」语义，只是把触发条件从「包版本变了」扩展到「环境变了」）。

**A5｜报告缺「范围之外」声明章节。**
调研文档 §4.5 专门警告：白皮书不覆盖基础设施验证，「不要把白皮书当基础设施资质依据引用」；§5 的四层模型里 AVIOR 只覆盖 (b) 层（第三方包）+ (a) 层的引用。PRD §1.4 对内边界画得很清楚，但 FR-BUNDLE-3 的报告结构里没有一节明确告诉审计员**这份证据不覆盖什么**：计算环境 IQ/OQ/PQ（申办方 IT 在 QMS 下的责任）、分析代码 QC/双重编程（(d) 层）、Part 11 控制（宿主系统责任）。审计辩护的常识是：不声明边界的证据会被按最大外延质疑。
**建议**：FR-BUNDLE-3 报告结构增加固定章节「范围与边界声明」：明确本证据包证明什么、不证明什么、各排除项的责任归属与依据（引调研文档 §4.5 / R-FDA.pdf / 白皮书原文）。同时 base/recommended 的默认豁免应在报告中**引用 R-FDA.pdf 作为出处**（四层模型 (a) 层），而不是无声跳过——否则「为什么这 30 个包没评估」是审计必问题。

**A6｜`decision: exclude` 的后置语义未定义。**
排除有两条路径：`scope.exclude`（评估前出范围，需报告说明）与决策 `exclude`（评估后拒绝）。后者的后果没有定义：被决策为 exclude 的包**仍在 renv.lock 里**时，`check` 应该红还是绿？绿=「我们决定不用它但它还装着」，审计上说不通；红=强制从 lockfile 移除，那 exclude 决策实际上是一个待办事项而非终态。
**建议**：定义为——decision `exclude` 的包若仍出现在 lockfile 中，`check` 红（缺陷类型：`excluded_but_present`），修复建议为「从项目移除该依赖，或改决策并说明」。把这一条写进 FR-CHECK-2 与 FR-REVIEW 的完整性规则。

### B 级：M1 开工前应决策

**B1｜默认权重与性能 AC 正面冲突：`covr_coverage` 是昂贵指标。**
§6.2 默认策略给 `covr_coverage` 权重 2.0（且 rationale 示例强调它最重要），但 riskmetric 的 covr 指标需要**安装包并完整跑一遍测试套件**——lme4 一个包就可能超过 30 分钟，与「50 包冷启动 ≤ 30 分钟」的 AC 冲突；调研文档 §3.2（valtools 缺点）也提醒过「报告编译含测试运行，复杂包可达数十小时」。这不是缓存能解决的（冷启动就会撞上）。
**建议**：(a) 适配层指标注册表（§7.2 `metrics()`）增加 `cost: metadata|network|execution` 标记，`execution` 级指标默认不在批量 assess 中运行，需 `--deep` 或按包显式开启；(b) 默认权重改用元数据可得的测试信号（riskmetric 的 `has_tests`/远端 check 结果类指标）承担「测试」准则，`covr_coverage` 作为 opt-in 深度指标；(c) W3 用真实 50 包 lockfile 做性能 spike，先于缓存开发校准 AC。另：样例 `avior.yml` 里的指标 id `bugs_status` 疑似不是 riskmetric 的真实指标名（其现役指标为 `last_30_bugs_status`）——M1 落适配器注册表时逐一核对样例，否则按「引用未注册指标 → check 报错」的规则，样例自己会红。

**B2｜评分缓存与 NA 的交互会产生「静默降级的分数」。**
FR-X-5 缓存键 =「包+版本+引擎+引擎版本+指标集」，不含**NA 结果集**。离线跑出的分（若干指标 NA、经 reweight 聚合）会被联网场景原样复用——同键不同质，且 reweight 后的分数看不出降级痕迹。FR-ASSESS-3 只要求 NA 在报告可见，没约束缓存行为。
**建议**：缓存条目记录 `na_metrics`；命中时若 `na_metrics` 非空且当前具备网络，视为可改进命中，默认重评（可配置）。同时报告逐包展示**生效权重向量**（reweight 后），让 QA 看得见每个分数是在什么指标可得性下算出来的。

**B3｜bundle 进 git 的体积策略缺失。**
Q3 决定 `evidence/` 入库（正确，git 提交哈希是信任锚点），但 FR-BUNDLE-1 同时产出同名 zip——zip 与目录内容重复，入库则体积翻倍且 zip 本身不 diff 友好（公理 A2 的张力）；docx 二进制同理但为签署流程所需，只能接受。长期按月/按发布出 bundle，仓库膨胀是确定事件。
**建议**：zip 定义为**传输件而非归档件**：默认不生成、`--zip` 时生成且默认 gitignore（zip 可由目录 + SOURCE_DATE_EPOCH 确定性重建，manifest 覆盖的是目录内容）。在 NFR 或用户手册中写明证据留存策略：git 历史即归档，工作区可裁剪旧 bundle（历史仍在），重签署场景从 QMS 归档取。

**B4｜决策过期粒度：V1 全量重审可接受，但 schema 要为 V2 留口。**
FR-REVIEW-3 规定版本变更即决策过期——方法论上安全，但意味着**任何 patch 升级都触发该包全量重决策**。这会是 design partner 试点期第一个抱怨点；J3（diffify 变更影响评估）在 V2 才来。若届时发现决策 schema 不支持「引用前一决策的增量评估记录」，就是破坏性变更。
**建议**：V1 行为不变（严格是对的），但决策 schema 现在就预留 `supersedes: <decision-ref>` 与 `assessment_type: initial|delta` 字段（default initial，V1 不使用）——与 FR-REVIEW-5 为 AI 双署名「schema 先行」完全同构的手法。

**B5｜NFR-5 dogfooding 混合了两件事，需拆开表述。**
「AVIOR 仓库自身跑 avior 出证据包」验证的是**管线**（对象是 AVIOR 的依赖）；「AVIOR 在自身默认策略下落入低风险档」验证的是**AVIOR 作为被评估包的风险画像**（对象是 avior 自己，需用引擎对其打分，这不是 avior 管线的常规输出——avior 评的是项目的依赖，不是项目本身）。两个目标都对，但混在一条 NFR 里，验收时会发现第二件事需要单独的 CI 作业。
**建议**：拆为 NFR-5a（管线 dogfooding：avior 跑自身开发依赖，CI 出证据包）与 NFR-5b（自身风险画像：CI 中直接以 riskmetric/默认策略给 avior 包打分，落低风险档）。

### C 级：路线级关注项

**C1｜北极星指标在零外发架构下不可全局观测。**
§10-5 已经比 ValiR 版诚实得多（不数「签署数」），但「通过完整性校验的证据包生成数」在 NFR-4 默认零外发的约束下，工具方仍然看不到全局值——它只在单项目内可观测。不要为了指标加 telemetry（会伤害这个产品在目标行业的信任根基）。
**建议**：把指标口径明确为「试点期 design partner 项目内统计 + 开源后以 GitHub 采纳信号（clone/star/issue 中的真实使用报告）为代理」，写进 §10 的度量说明即可，不改产品。

**C2｜val.meter 迁移窗口。**
风险表已覆盖，补一个可执行动作：M1 落 riskmetric 适配器时，同步写**引擎适配层契约测试**（对注册表/assess 返回结构的 golden 测试），让「新引擎适配器是否合格」变成跑一遍测试套件的事。W12 决策点加一项：核查 val.meter 成熟度（CRAN 状态、指标覆盖），决定 V2 是否上适配器。

**C3｜文档基线的陈旧引用（房务级）。**
根 README（中英双栏）徽章与正文仍写「PRD v1.2」（现为 v1.5）；根 README 仓库结构与 `docs/README.md` 索引仍引用 `r-package-validation-strategy.docx`，该文件已被 `.md` 版取代（commit 7685c2d），索引链接已断。受控文档体系里索引与基线号不一致，是审计观感问题。
**建议**：单独一个小 PR 修正；顺带在 README 徽章处考虑改为不带版本号的「PRD: development baseline」链接，避免每次修订都要追改。

## 3. 路线与里程碑评审

当前时点（2026-07-12）处于 W1 末：M0 样例已组装完成，**但 M0 的出口标准是「design partner 的 QA 审阅样例后确认可进入签署流程」——这一步尚未发生，且它是全项目最大的假设检验**。样例中的 A2 不一致应在送审前修掉，避免把错误格式送进 QA 评审。

对 M1–M3 的三个判断：

1. **12 周对 1–2 人是紧的，但架构够薄、可行**；前提是 M1 不返工。返工的最大来源就是 §2-A 级问题——都是几行契约文字，冻结前改零成本。
2. **报告模板是进度黑洞的传统位置**（Quarto→docx 的样式打磨、中文排版、GAMP 5 叙事结构与 QA 意见的往返）。建议 M2 内部排序：HTML 报告先行打通全链路（依赖最轻，风险表自己也这么判断），docx 模板与 QA 意见回灌放 M3 试点期迭代，不要让 docx 样式阻塞 J1 全旅程跑通。
3. **三平台字节一致是横切验收，无法后补**。若 M1 用「先跑通再说」的方式写序列化，M3 收敛三平台 CI 时会付出数倍代价。golden 文件快照测试 + 规范化写出器（A1）必须是 M1 第一周的基建。

## 4. 下一阶段开发指引

### 4.1 本周（M0 收尾，W2）

1. **修 A2**（`version_managed` 契约二选一），同步 PRD §6.5 与样例 CSV；顺手把 A5 的「范围与边界声明」章节加进样例 `report.html`，让 QA 评审的就是最终形态。
2. **落 A1/A3/A4/A6 为 PRD 修订 PR**（v1.6）：FR-X-8 规范化序列化、FR-BUNDLE-5 环境指纹扩展、test-results 环境字段、exclude 后置语义。B4 的 schema 预留字段一并进。
3. **把样例送 QA 评审**（M0 DoD），评审意见以 issue 落档、回灌 PRD/样例。
4. C3 房务 PR（README 版本引用与断链）。

### 4.2 M1 启动（W3–6）

**第一周基建（顺序敏感）**：
- R 包骨架：`usethis` 脚手架、R ≥ 4.1 底线、核心依赖锁定（cli/yaml/jsonlite 级）、testthat 3e、lintr/styler、GitHub Actions 三平台矩阵**从第一个 commit 就开**。
- **规范化写出器先于一切命令**：实现 FR-X-8 的 YAML/CSV/JSON 写出工具（编码、换行、数字格式、键序、排序），配 golden 快照测试；此后所有产物必须经它落盘。
- **性能 spike（B1）**：拿 design partner 真实 50 包 lockfile 直接裸跑 riskmetric，实测冷启动耗时与逐指标成本，据此定指标注册表的 `cost` 分级与 assess 默认指标集，回填 §8 目标值。这个 spike 半天，比等到 W6 撞 AC 便宜得多。

**命令实现顺序**（每个命令的产物校验器同时就是未来 `check` 的规则，边写边攒）：
1. 配置加载 + schema 校验（`avior.yml`，含 rationale TODO 检查）；
2. `scan`：renv.lock 解析 → 三分类 → 静态调用识别（`library()`/`require()`/`::`）→ inventory 写出；用样例项目做端到端 golden 测试；
3. 引擎适配层接口 + riskmetric 适配器 + **适配层契约测试**（C2）；核对样例中全部指标 id；
4. `assess` + 缓存（含 B2 的 na_metrics 记录）；
5. `review` 决策桩生成 + 完整性校验（含 A6 规则）。

**并行事项**：
- **dogfooding 提前**：scan/assess 一可用就在 AVIOR 自己的 CI 里跑（NFR-5a），不要等 M3——它是最早的真实用户。
- **方法论对照文档（NFR-9）增量写**：每落一个命令，补一段「该产物对应白皮书/GAMP 5 哪一条」，M3 只做汇编不做创作。
- FR-SCAN AC 的「识别差异 ≤ 10%」度量脚本随 scan 一起写好，W6 用真实项目跑数。
