> **本 PRD 的前置假设（需在评审中确认）**
> 1. 本产品作为**对外商业产品**发布（SaaS + 私有化/本地部署双形态），而非纯内部工具。
> 2. 首要目标客户是**制药/生物技术/CRO 的临床统计编程与数据科学团队**，其次是任何在受监管环境用 R 的团队。
> 3. 产品**不替客户承担合规责任**——它是"证据产出与编排引擎"，最终判断与签署始终由客户完成（这是法律与合规的硬边界）。
> 4. AI 能力用于**降低人工负担**（起草需求/测试/SOP/阈值建议），但所有 AI 产物必须经人工复核后才能进入证据链。

## 1. 背景与问题陈述

### 1.1 市场背景

FDA、EMA、NMPA 均不禁止在监管申报中使用 R 与开源包，但要求申办方为任何用于产生申报结论的软件建立"与风险相称、可追溯、可复现"的验证证据链。随着 R 在临床统计编程中取代/补充 SAS，"如何合规地验证 R 包"已从边缘话题变成申报级刚需。

当前市场存在四类玩家（详见竞品分析）：开源标准/框架派（R Validation Hub、PHUSE valtools）、商业验证包订阅派（Atorus OpenVal）、咨询+工具派（Jumping Rivers）。**但没有一家提供覆盖"自研包 + 第三方包 + 环境 + 持续监控 + 证据链 + 审计签署"的一体化产品。**

### 1.2 核心痛点

团队要落地一套合规的 R 包验证体系，今天必须自己缝合至少 7 个独立工具，并承担大量人工编排工作：

1. **工具链碎片化**：riskmetric（第三方打分）、riskassessment（评审）、valtools（自研脚手架）、PPM（环境）、renv（锁定）、diffify（版本差异）、oysteR（CVE）分属不同项目、不同界面、不同数据格式，无统一编排。
2. **人工负担重**：需求（URS/FS）、测试用例、testthat 测试代码、风险阈值/权重、SOP 均需人工从零编写，是最大的时间黑洞。
3. **准确性验证缺失**：纯自动化打分工具（riskmetric）明确不判断"算得对不对"，高风险统计包仍需大量人工深度测试。
4. **证据链难以自动串起**：RA → 测试 → 追溯矩阵 → 变更控制 → 电子签署的完整链条需手工拼接，审计时易断裂。
5. **可持续性风险**：riskmetric 转维护、val.meter 演进中，团队自建易被上游变动打断。
6. **双标准/国际化**：国内外申报证据链需可复用，但中文 SOP、签署记录、归档需并行维护。

### 1.3 产品机会

把上述碎片化工作整合为**单一平台上的端到端流水线**，并用 **AI 补齐人工瓶颈**（自动起草需求/测试/SOP、给出阈值建议），同时以**审计追踪 + 电子签名（21 CFR Part 11 / Annex 11）**守住合规证据链。这既非纯自建（重复造轮子），也非纯采购（锁定与覆盖不全）——而是把"自建为主 + 复用开源 + AI 提效"打包成可交付的产品。

## 2. 产品愿景与目标

### 2.1 愿景

> 让任何在受监管环境使用 R 的团队，都能像使用现代 CI/CD 一样，把"合规验证"变成一条可复现、可追溯、可审计、大部分自动化的流水线。

### 2.2 产品目标（12 个月内）

- **G1｜降本**：把一个中等规模项目（约 50 个第三方包 + 5 个自研包）从"首次验证就绪"的人工工时降低 ≥ 60%。
- **G2｜合规**：产出的证据链可直接支撑 FDA/EMA/NMPA 申报与审计，通过至少 3 家客户的内部 QA/审计验收。
- **G3｜整合**：单平台覆盖两条验证主线 + 环境 + 持续监控 + 证据链，替代 ≥ 5 个独立工具的手工编排。
- **G4｜可采纳**：MVP 上线 6 个月内获得 ≥ 10 家付费客户 / ≥ 30 个活跃项目。

### 2.3 北极星指标与关键指标

| 类型 | 指标 | 说明 |
| --- | --- | --- |
| 北极星 | **每月"经签署归档的验证证据包"数量** | 直接反映平台产出合规价值的核心 |
| 采纳 | 活跃项目数、周活团队数 | |
| 效率 | 单包平均验证工时、AI 采纳率（AI 草稿被采纳/编辑后采纳的比例） | |
| 质量 | 审计发现（findings）数、证据链完整性校验通过率 | |
| 商业 | 付费客户数、NRR（净收入留存）、从试用到付费转化率 | |

**差异化护城河（三条）**
1. **唯一一体化双线平台**：把两条验证主线 + 环境 + 持续监控 + 证据链装进一个界面、一套数据模型。
2. **AI 提效层**：直击文档反复点出的"人工写需求/测试/SOP"最大瓶颈，且保留 human-in-the-loop 合规边界。
3. **开源友好、反锁定**：底层编排成熟开源工具而非黑盒，证据可导出、可迁移，SOP 与具体工具解耦（只绑定"四准则"）。

## 5. 产品范围

### 5.1 MVP（第一版，聚焦"跑通小闭环"）

对应文档的"最小可行起步"，MVP 必须能让客户用一个真实项目跑通：第三方包评分 + 自研包一次完整验证 + 环境冻结 + 一份可签署归档的证据包。

- M-A 项目与 SOP 配置（基础版）
- M-B 第三方包风险评估流水线（集成 riskmetric）
- M-C 自研包 SDLC 验证（集成 valtools + testthat + covr）
- M-D 环境快照与可复现（集成 PPM/renv）
- M-E AI 辅助 v1（需求草稿、测试用例草稿、阈值建议）
- M-F 证据链 + 审计追踪 + 电子签名
- M-G 报告导出（PDF/Word/追溯矩阵）

### 5.2 Fast-Follow（V2）

- 持续监控与变更控制引擎（diffify / oysteR / 定时再评估触发）
- 多项目/多团队治理、模板库、SOP 版本管理
- 中英双语 SOP 与归档、NMPA 视角合规包
- CI/CD 深度集成（GitHub Actions / GitLab CI 一键接入）

### 5.3 后续（V3+）

- "已验证包"跨组织共享市场（对标 OpenVal 的社区效应，但以联邦/隐私保护方式）
- 高风险统计包"金标准比对"库（PK、生存分析等）
- 外援市场：一键对接第三方咨询/单包验证采购
- 面向 Shiny 应用、Python/其他语言的验证扩展

### 5.4 明确不做（Out of Scope，MVP 阶段）

- 不替客户做合规判断或签字（法律边界）
- 不做通用 CRAN 镜像/包管理器（用 PPM，不重造）
- 不承接人工咨询服务交付（可作为合作伙伴转介）

## 7. 关键用户旅程

**旅程 1｜首次为一个项目建立验证体系（对应文档"最小可行起步"）**
1. 创建项目 → AI 起草 SOP 骨架与阈值建议 → 负责人确认固化（模块 A/E）。
2. 接入代码仓库 → 自动依赖分析，识别 intended-for-use 直接调用包（模块 C）。
3. 一键 riskmetric 批量打分 → 评审界面人工评级/决策（模块 C）。
4. 选 1 个自研包接入 valtools → AI 草拟需求与测试 → 人工复核 → 跑 testthat/covr → 生成验证报告（模块 D/E）。
5. 冻结环境（PPM/renv）→ 产出可复现性证据（模块 B）。
6. QA 审查 → 多角色电子签署 → 导出审计包归档（模块 G）。

**旅程 2｜包版本升级触发再验证**
版本变更被 diffify 捕获 → AI 总结差异影响 → 触发变更控制工作流 → 定向再验证 → 签署更新证据链（模块 F/E/G）。

## 8. 非功能需求

| 类别 | 需求 |
| --- | --- |
| **合规** | 21 CFR Part 11、EU Annex 11 电子记录/签名；GAMP 5 交付物；ALCOA+ 数据完整性；支持 NMPA 中文归档与签署留痕 |
| **安全** | 客户数据隔离、传输/静态加密、RBAC、SSO/SAML、密钥管理；本地部署形态支持完全离线 |
| **审计追踪** | 全量、不可篡改、时间戳可信、可导出；AI 交互同样纳入 |
| **可复现** | 环境指纹与哈希、快照留存、一键复现校验 |
| **性能** | 批量打分 500+ 包可在合理时间完成；自研包报告编译支持并行与缓存以规避"数十小时"问题 |
| **可用性** | 面向统计程序员与 QA 的双重界面友好度；关键操作 ≤ 3 步 |
| **可扩展/反锁定** | 评估引擎、环境后端、CI 均为适配层，可替换；证据可完整导出迁移 |
| **部署** | 云 SaaS 与私有化/本地部署（受监管客户常要求 on-prem/VPC） |
| **国际化** | 中英双语界面、SOP 与报告模板 |

## 9. 技术架构概览（示意）

```
┌─────────────────────────────────────────────────────────────┐
│  前端：项目控制台 / 评审界面 / 风险看板 / 审计只读视图         │
├─────────────────────────────────────────────────────────────┤
│  编排层（核心）：验证工作流引擎 · 状态机 · 触发引擎           │
├───────────────┬───────────────┬───────────────┬─────────────┤
│ 第三方评估适配 │ 自研验证适配   │ 环境适配       │ 监控适配     │
│ riskmetric/   │ valtools/     │ PPM/renv      │ diffify/     │
│ val.meter     │ testthat/covr │               │ oysteR/osv   │
├───────────────┴───────────────┴───────────────┴─────────────┤
│  AI 服务层：需求/测试/SOP 生成 · 影响分析（human-in-loop）    │
├─────────────────────────────────────────────────────────────┤
│  证据链与合规：不可篡改存储 · 审计追踪 · 电子签名 · 追溯图谱  │
├─────────────────────────────────────────────────────────────┤
│  基础设施：RBAC/SSO · 加密 · 多租户/单租户 · 云/本地部署      │
└─────────────────────────────────────────────────────────────┘
```

**设计原则**
- **适配层解耦**：所有开源工具通过适配器接入，SOP 只绑定"四准则"方法论，屏蔽 riskmetric 转维护、val.meter 演进等上游变动。
- **证据不可变**：证据链一旦签署即冻结版本，任何变更走新版本 + 变更控制。
- **AI 有边界**：AI 产物默认未确认，人工确认才入链，全程可审计。

## 12. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 上游开源工具变动（riskmetric 转维护、val.meter 演进） | 集成失效、指标口径变 | 适配层解耦；SOP 只绑四准则；跟踪并预置 val.meter 适配 |
| 合规责任边界被误解为"产品替客户合规" | 法律与信任风险 | 产品定位反复强调"仅产证据"；AI 不签署；文档与合同明确 |
| AI 生成内容质量不足或产生合规隐患 | 证据链可信度受损 | 强制 human-in-the-loop、双署名、全程审计、可复盘 |
| 电子签名/审计追踪不达 Part 11 | 无法用于申报 | 合规需求前置设计，第三方合规评估背书 |
| 客户要求本地/离线部署 | 交付复杂度高 | 架构从一开始支持单租户/本地形态 |
| 与现有开源工具"重复"被质疑价值 | 采纳阻力 | 突出"一体化编排 + AI 提效 + 证据链"三点单一工具无法提供的价值 |
| 报告编译耗时（复杂包数十小时） | 里程碑延误、体验差 | 分层/并行/缓存/关键包优先 |

## 13. 成功标准（Definition of Success）

- MVP 能让 design partner 在 **2 周内**跑通一个真实项目的"小闭环"，并产出**一份可通过内部 QA 的可签署证据包**。
- 中等项目首次验证就绪人工工时**下降 ≥ 60%**。
- 至少 **1 家客户**用平台产出的证据链**支撑一次真实申报或内部审计并通过**。
